🔧 디버깅 팁 및 유지보수

📊 1-1. 전체 에러율 확인하기

대시보드 상단의 통계로 에러 발생 여부를 빠르게 파악합니다
❌ 문제 상황
평소에는 잘 작동하던 워크플로우에서 갑자기 고객 알림이 가지 않는다는 제보가 들어왔습니다. 무엇부터 확인해야 할까요?
✅ 해결 방법
대시보드 상단 통계 확인
📈 체크 포인트
지표 정상 상태 문제 의심
에러율 0-5% 10% 이상 급증
실패 횟수 평소와 유사 갑작스런 증가
실행 간격 규칙적 중단되거나 불규칙

📋 1-2. 에러가 발생한 실행 확인하기

Executions 목록에서 실패한 실행을 찾아 상세 정보를 확인합니다
🔍 실행 목록 정보
  • 상태 아이콘 - 성공(초록), 실패(빨강), 대기(회색)
  • 실행 시간 - 언제 실행되었는지
  • 워크플로우 이름 - 어떤 워크플로우인지
  • 실행 모드 - 자동/수동 실행 여부
💡 빠른 필터링 팁
  • 상태 필터 - "Error"만 선택하여 실패 건만 보기
  • 시간 범위 - 최근 24시간, 일주일 등
  • 워크플로우 필터 - 특정 워크플로우만 보기
  • 정렬 - 최신순으로 정렬하여 최근 에러 우선 확인

🎯 1-3. 에러가 발생한 노드 확인하기

실행 상세 화면에서 어느 노드에서 에러가 발생했는지 찾습니다
🔴 에러 노드 특징
  • 빨간색 테두리 - 에러 발생 노드
  • 느낌표 아이콘 - 경고 표시
  • 데이터 없음 - 출력 데이터가 비어있음
  • 이후 노드 회색 - 실행이 중단됨
💡 노드별 에러 예시
노드 종류 자주 발생하는 에러
HTTP Request API 연결 실패, 타임아웃
Code 문법 오류, undefined 접근
Gmail 인증 만료, 잘못된 이메일
Set 존재하지 않는 필드 접근

💬 1-4. 에러 메시지 확인하기

에러 메시지를 읽고 근본적인 원인을 파악합니다
❌ 1단계: 에러 메시지 확인
에러 메시지를 확인한 결과:

Upstage Solar LLM failed for item 0: Node 'When chat message received' hasn't been executed

파악된 문제: 실행하지 않은 노드인 'When chat message received'를 참조해서 이슈가 발생함
🔍 2단계: 구체적 원인 확인
'Upstage Solar LLM' 노드의 설정을 확인한 결과:

문제 위치: 'Content' 부분 파라미터에서 실행되지 않은 'When chat message received' 노드의 데이터를 참조하고 있음

해결 방법: 해당 노드를 먼저 실행하거나, 올바른 노드를 참조하도록 수정 필요

🔔 1-5. 에러 발생 시 알람 받기

에러가 발생하면 즉시 알림을 받아 빠르게 대응할 수 있습니다
1️⃣ Error Trigger 워크플로우 생성
설정 방법:
워크플로우 캔버스에 'Error Trigger' 노드를 추가하여 실제 문제가 발생하면 자동으로 작동하도록 설정합니다.

역할: 다른 워크플로우에서 에러가 발생할 때마다 이 워크플로우가 실행됩니다.
2️⃣ 알림 내용 설정 (Slack 등)
전달할 핵심 정보:
• 에러 설명 (어떤 문제가 발생했는지)
• 에러 발생한 워크플로우 URL
• 실패한 노드 이름
• 발생 시간

Slack, Discord, 이메일 등 메신저를 통해 즉시 알림을 받을 수 있습니다.

📌 2-1. 핀 고정으로 데이터 고정하기

같은 데이터로 반복 테스트할 때 가장 유용한 디버깅 도구입니다
1️⃣ 핀을 달 노드 확인
에러가 발생한 위치를 확인하여 어느 노드에 핀을 달아야 하는지 파악합니다.

에러 발생 이전의 노드 데이터를 고정하여 반복 테스트에 활용합니다.
2️⃣ 핀 아이콘 클릭
첫 번째 노드에 들어가서 우측 상단의 핀 📌 아이콘을 클릭합니다.

이렇게 하면 해당 노드의 출력 데이터가 고정됩니다.
3️⃣ 데이터 보존 확인
에러가 발생한 노드를 들어가더라도 앞에 있던 노드의 값이 보존됩니다.

이제 API를 다시 호출하지 않고도 반복 테스트가 가능합니다.

▶️ 2-2. Manual Trigger로 시작 지점 변경하기

테스트할 특정 워크플로우 구간으로 시작 지점을 바꿔서 빠르게 실험합니다
💡 Manual Trigger 활용법
시작 지점 변경:
테스트를 수행할 특정 워크플로우 구간에 Manual Trigger 위치를 바꿔서 (즉, 시작 지점을 바꿔서) 실행을 시킵니다. 이렇게 하면 해당 부분부터 다시 실험을 진행할 수 있습니다.

🔥 핀(Pin)과 함께 활용하면 시너지가 높습니다!
• 핀 고정으로 이전 노드의 데이터를 보존하고
• Manual Trigger로 시작 지점을 특정 구간으로 설정
• 결과: 전체 워크플로우를 실행하지 않고도 원하는 부분만 빠르게 반복 테스트 가능!

예시: 10개 노드 중 7번째 노드에 핀 고정 → 8번째 노드에 Manual Trigger 연결 → 8~10번 노드만 빠르게 테스트

🎯 2-3. 특정 노드만 개별 실행하기

제한적인 범위에서만 실행하여 더 빠르게 디버깅할 수 있습니다
🔄 Manual Trigger vs 노드 실행 버튼
방식 실행 범위 설명
Manual Trigger 시작 지점 → 끝까지 해당 위치부터 워크플로우 끝까지 실행
노드 실행 버튼 Pin 위치 → 해당 노드까지 해당 노드까지만 실행 (더 제한적)
🔥 Pin + 노드 실행 시너지
Pin을 앞에 붙여두는 경우:

Pin 위치부터 해당 노드까지만 실행되기 때문에 제한적인 범위에서의 실행이 가능합니다.

예시:
• 3번 노드에 Pin 고정
• 5번 노드 선택 후 실행 버튼 클릭
• 결과: 3번 → 4번 → 5번 노드만 실행
• 1, 2번 노드와 6번 이후 노드는 실행 안 됨

→ 가장 빠르고 정확한 디버깅 가능!

⌨️ 2-4. 텍스트 인풋 주는 방법

외부 트리거가 필요한 경우 Chat Trigger와 Set 노드로 테스트 환경을 구축합니다
💡 Chat Trigger로 테스트하기
사용 상황:
텍스트 인풋이 들어가야 하는데 외부에서 Trigger가 되어야 하는 경우, 테스트 용이성을 위해 Chat Trigger를 달아 텍스트를 대신 인풋으로 넣어 실험을 진행합니다.

장점:
• 실제 외부 트리거 없이도 빠른 테스트 가능
• 다양한 입력값으로 즉시 실험
• 실제 사용자 시나리오 재현
🔥 Set 노드로 변수 동일하게 세팅
핵심 전략:
Chat Trigger 다음에 Set 노드를 중간에 두어 외부에서 Trigger되어 들어오는 변수와 동일하게 세팅합니다.

효과:
뒷 노드들의 값을 수정하지 않고 바로 테스트를 진행할 수 있습니다!

예시:
실제 Webhook은 body.userId를 전달하지만,
Set 노드에서 body.userId와 동일한 구조로 세팅하면
→ 뒷단 노드들은 수정 없이 그대로 사용 가능
📋 구성 예시
[워크플로우 구조] Chat Trigger (테스트용) ↓ Set 노드 (외부 트리거 변수 형식과 동일하게 세팅) { "body": { "userId": "test123", "message": "주문 조회해줘" } } ↓ 실제 비즈니스 로직 노드들... ({{ $json.body.userId }} 등 그대로 사용 가능)

🤖 3-1. GPT를 활용한 문제 해결

에러 메시지와 상황을 GPT에게 설명하여 빠르게 해결책을 찾습니다
✅ GPT 활용 전략
📝 GPT에게 전달할 핵심 정보
1️⃣ n8n이라는 상황 전달
"n8n 워크플로우에서..." 로 시작하여 GPT가 n8n 환경임을 인지하도록 합니다.

2️⃣ 어떤 결과를 얻고 싶은지 알려주기
"API 응답 데이터를 배열로 가공하고 싶습니다" 등 원하는 결과를 명확히 설명합니다.

3️⃣ 어떤 문제 상황인지 알려주기
가능하면 에러 메시지를 복사해 붙여넣어 문제를 구체적으로 알려줍니다.
예: TypeError: Cannot read property 'map' of undefined

🔄 3-2. Continue on Error 설정하기

에러 발생 시 워크플로우를 어떻게 처리할지 선택할 수 있습니다
⚙️ Continue on Error 설정
📋 3가지 에러 처리 옵션
1️⃣ Stop Workflow
에러가 발생하면 Workflow를 멈춥니다.
활용: 뒤 워크플로우가 앞 작업의 값을 받아 진행할 필요가 있는 경우

2️⃣ Continue
에러가 발생해도 무시하고 뒤 워크플로우가 작동합니다.
활용: 에러가 잦고 영향이 낮은 작업이나 병렬적인 작업 중 하나인 경우

3️⃣ Continue (using error output)
에러가 발생한 경우 이 노드를 별도로 처리해 주고 갑니다.
활용: 특히 Loop 문에서 에러 발생 건을 처리해 버리는 경우에 주로 활용

🎯 3-3. 올바르게 데이터 접근하는 방법

n8n 데이터 구조를 이해하고 올바르게 참조하면 대부분의 에러를 예방할 수 있습니다
📊 n8n 데이터 접근 Hierarchy
graph TD A[워크플로우] --> B[노드 Node] B --> C[Items 배열] C --> D[Item 0] C --> E[Item 1] C --> F[Item 2...] D --> G[json] D --> H[binary] G --> I[필드 Fields] I --> J[name] I --> K[email] I --> L[age...] style B fill:#E5F5FF,stroke:#1B365D,stroke-width:2px style C fill:#FFFAF0,stroke:#DD6B20,stroke-width:2px style G fill:#F0FFF4,stroke:#38A169,stroke-width:2px style I fill:#FFF5F5,stroke:#E53E3E,stroke-width:2px
접근 순서: Node → Items → json/binary → Fields
1️⃣ 바로 앞의 노드를 받는 경우
// 바로 앞 노드의 데이터 접근 {{ $json.xx }} // 예시 {{ $json.name }} {{ $json.user.email }} {{ $json.items[0] }}
사용 상황:
직전 노드의 출력을 받아서 사용할 때

특징:
• 가장 간단하고 자주 사용하는 방법
• $json으로 시작
• 노드 이름 지정 불필요
2️⃣ 특정 위치의 노드를 받는 경우
// 노드 명칭부터 기록하기 {{ $("노드명").item.json.xx }} {{ $("노드명").items[0].json.xx }} // 예시 {{ $("HTTP Request").item.json.userId }} {{ $("Set").items[0].json.name }} {{ $("When chat message received").item.json.chatInput }}
사용 상황:
여러 단계 이전의 노드나 특정 노드의 데이터를 참조할 때

특징:
• 노드 명칭을 정확히 지정
• .item.json 또는 .items[0].json 사용
• 워크플로우 구조가 복잡할 때 유용
💡 실전 팁: 올바른 예시 vs 자주 하는 실수
✅ 올바른 예시:

• {{ $json.email }}
→ 바로 앞 노드의 email 필드

• {{ $json.user.name }}
→ 중첩된 객체 접근

• {{ $("HTTP Request").item.json.data }}
→ 특정 노드의 data 필드

• {{ $("Set").items[0].json.result }}
→ 배열의 첫 번째 아이템
❌ 자주 하는 실수:

• {{ json.email }}
→ $ 가 빠짐

• {{ $json }}
→ 필드명이 빠짐

• {{ email }}
→ $json이 빠짐

• {{ $("노드명").json.xx }}
→ .item 또는 .items[0]이 빠짐

• {{ $node["노드명"].json }}
→ 구 문법 (새 문법: $("노드명") 사용)
🎯 핵심 원칙:
1️⃣ 바로 앞 노드: {{ $json.필드명 }}
2️⃣ 특정 노드: {{ $("노드명").item.json.필드명 }}
3️⃣ 항상 확인: 노드 실행 후 출력 데이터 구조를 먼저 확인하고 참조하기!

→ 에러율, 실패 횟수 증가 여부 파악
Executions 목록에서 실패 건 확인
→ 빨간색 실행 클릭, 언제부터 발생했는지 추적
빨간색 에러 노드 찾기
→ 어느 노드에서 문제가 발생했는지 위치 파악
에러 메시지 정확히 읽기
→ 에러 내용 복사, 근본 원인 이해
Error Trigger 알림 설정
→ Slack/Discord로 즉시 알림 받도록 구성
🧪 2단계: 문제 재현
핀 고정 (📌)
→ 우측 상단 핀 아이콘으로 데이터 보존
→ API 재호출 없이 반복 테스트
Manual Trigger 활용
→ 시작 지점을 테스트 구간으로 변경
→ 해당 위치부터 끝까지 실행
특정 노드만 실행
→ 노드 선택 후 실행 버튼
→ Pin 위치 ~ 해당 노드까지만 실행
Chat Trigger + Set 노드
→ 외부 트리거 없이 테스트 데이터 입력
→ Set으로 변수 구조 동일하게 세팅
🛠️ 3단계: 문제 해결
GPT에게 질문
• n8n 상황 전달
• 원하는 결과 설명
• 에러 메시지 복사 붙여넣기
Continue on Error 설정
• Stop Workflow: 앞 작업 의존
• Continue: 영향 낮은 작업
• using error output: Loop 에러 처리
올바른 데이터 참조
• 바로 앞: {{ $json.xx }}
• 특정 노드: {{ $("노드명").item.json.xx }}
🔥 핵심 디버깅 시너지 조합
조합 1: Pin + 노드 실행 버튼
→ 제한적 범위에서 가장 빠른 테스트
→ 예: 3번 노드 Pin → 5번 노드 실행 → 3~5번만 실행

조합 2: Pin + Manual Trigger
→ 데이터 보존 + 시작 지점 변경
→ 전체 워크플로우 실행 없이 특정 구간만 반복 테스트

조합 3: Chat Trigger + Set + Pin
→ 외부 트리거 시뮬레이션
→ Set으로 변수 구조 통일 → 뒷 노드 수정 불필요

조합 4: Error Trigger + Continue on Error
→ 에러 발생해도 계속 진행 + 즉시 알림
→ 서비스 중단 방지 + 빠른 대응
💡 실전 디버깅 시나리오별 대응법
상황 추천 기법 핵심 포인트
API 에러 반복 발생 Pin + 노드 실행 API 재호출 없이 빠른 수정 테스트
데이터 참조 에러 GPT + 올바른 참조 $json.xx 또는 $("노드명").item.json.xx
Loop 내 일부 실패 Continue (using error output) 실패 건만 별도 처리, 나머지 계속 진행
외부 Webhook 테스트 Chat Trigger + Set 외부 트리거 없이 동일 구조로 테스트
복잡한 워크플로우 Manual Trigger 이동 시작 지점 변경으로 특정 구간만 테스트
🎓 디버깅 마스터의 핵심 원칙:

5️⃣ 체계적 접근 - 발견 → 재현 → 해결 3단계 프로세스 준수
