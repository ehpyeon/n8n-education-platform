🔧 24. 에러 디버깅

문제 해결 전문가를 고용해서 완벽한 자동화 시스템을 만듭니다

  1. 실제 에러 사례 분석 - 고객 문의, 매출 데이터, 주문 처리 오류 해결
  2. 실행 이력 추적 기법 - 문제 발생 지점을 정확히 찾는 방법
  3. 핀 고정 디버깅 - 동일 조건에서 반복 테스트하는 기술
  4. Try/Catch 에러 방지 - 워크플로우가 멈추지 않도록 보호하는 방법

🚨 에러 디버깅이 중요한 이유

완벽한 자동화 시스템도 예상치 못한 상황에서 문제가 발생할 수 있습니다
💰 비즈니스 영향
  • 고객 문의 147건 6시간 미처리
  • 3주간 잘못된 매출 데이터 보고
  • 주문 확인 메일 30% 발송 실패
  • 고객 불만 및 매출 손실 발생
⚡ 신속한 대응의 가치
  • 실행 이력으로 30분 내 원인 파악
  • 핀 고정으로 정확한 문제 재현
  • Try/Catch로 사전 에러 방지
  • 모니터링으로 실시간 문제 감지
핵심: 에러를 빠르게 발견하고 해결하는 능력이 자동화 시스템의 신뢰성을 결정합니다.

📞 사례 1: 고객 문의 시스템 마비

외부 API 변경으로 인한 JSON 파싱 에러 사례를 살펴봅니다
🔍 문제 상황
항목 내용
증상 Slack 알림이 오지 않음
원인 API 응답 형식 변경
피해 6시간 동안 147건 미처리
해결시간 30분 내 복구
🛠️ 해결 과정
  1. 실행 이력 확인
    • Executions 탭에서 실패한 실행 찾기
    • 빨간색 에러 노드 확인
  2. 에러 메시지 분석
    • "JSON parsing failed" 에러 발견
    • API 응답 구조 변경 확인
  3. JSON 구조 수정
    • Set 노드에서 JSON 경로 업데이트
    • 테스트 실행으로 검증

📊 사례 2: 매출 데이터 집계 오류

데이터 단위 변경으로 인한 계산 오류와 핀 고정을 활용한 해결법을 다룹니다
📈 문제 분석
기간 예상 매출 계산된 매출 오차율
1주차 500만원 5,000만원 10배
2주차 750만원 7,500만원 10배
3주차 620만원 6,200만원 10배
🔧 핀 고정 활용 해결
  1. 과거 데이터 핀 고정
    • 정상 작동했던 실행 이력 선택
    • HTTP Request 노드에 핀 고정
  2. 계산 로직 검증
    • Set 노드에서 단위 확인
    • 가격 × 수량 계산식 점검
  3. 단위 변경 반영
    • '10원' 단위를 '원' 단위로 수정
    • 과거 3주 데이터 재계산

🔍 실행 이력으로 에러 추적하기

모든 워크플로우 실행 기록을 활용해 문제 발생 지점을 정확히 찾는 방법입니다
📝 실행 이력 확인 방법
  1. 워크플로우 에디터에서 확인
    • 우측 패널 "Executions" 탭 클릭
    • 각 노드별 데이터 실시간 확인
    • 빨간색 에러 노드 즉시 식별
  2. 전체 실행 탭에서 분석
    • 상단 메뉴 "Executions" 이동
    • 시간순 실행 이력 확인
    • 실패한 실행만 필터링
🎯 에러 패턴 분석
에러 유형 발생 빈도 주요 원인
API 연결 실패 40% 네트워크 또는 인증 문제
데이터 파싱 오류 35% JSON 구조 변경
필수 필드 누락 25% 입력 데이터 불완전
실습 팁: 에러가 발생하면 즉시 실행 이력을 확인하는 습관을 만드세요. 90%의 문제는 여기서 원인을 찾을 수 있습니다.

📌 핀 고정 실행: 디버깅의 게임체인저

특정 노드의 데이터를 고정해서 반복 테스트할 수 있는 강력한 기능입니다
⚡ 핀 고정의 장점
  • 시간 절약 - 긴 워크플로우 앞부분 재실행 불필요
  • 안전한 테스트 - 실제 데이터 영향 없이 집중 테스트
  • 정확한 재현 - 동일한 데이터 조건에서 반복 가능
  • 효율적 디버깅 - 문제 구간만 집중 분석
🎯 핀 고정 활용 패턴
Webhook → HTTP Request → [Set 노드] ← 📌 핀 고정! ↓ [문제 노드들]
  1. 문제 직전 노드에 고정
  2. 에러 재현용 데이터 고정
  3. 다양한 시나리오 테스트

🛠️ 핀 고정 실습 가이드

실제 에러 상황에서 핀 고정을 활용해 문제를 해결하는 단계별 방법입니다
1️⃣ 정상 데이터 고정
  1. 정상 실행 선택
  2. HTTP Request 노드 클릭
  3. 📌 핀 아이콘 클릭
  4. 데이터 고정 확인
2️⃣ 에러 데이터 고정
  1. 실패한 실행 찾기
  2. 문제 노드 입력 데이터 고정
  3. 동일 조건 에러 재현
  4. 해결책 반복 테스트
3️⃣ 다양한 케이스 테스트
  1. 빈 데이터 케이스
  2. 특수 문자 포함 데이터
  3. 대용량 데이터 처리
  4. 경계값 테스트
주의사항: 핀 고정된 데이터는 실제 실행에 영향을 주지 않으므로, 테스트 완료 후 핀을 해제하는 것을 잊지 마세요.

🛡️ Try/Catch로 에러 방지하기

에러가 발생해도 워크플로우가 완전히 멈추지 않도록 보호하는 필수 기능입니다
🎯 Try/Catch를 사용하는 이유
  • 워크플로우 중단 방지 - 한 부분 에러로 전체 중단 방지
  • 대체 처리 가능 - 에러 시 다른 방법으로 계속 진행
  • 에러 알림 - 담당자에게 즉시 문제 상황 알림
  • 복구 전략 - 자동으로 재시도하거나 백업 방법 실행
📊 Try/Catch 효과
상황 Try/Catch 없이 Try/Catch 적용
API 연결 실패 워크플로우 중단 대체 API 사용
이메일 발송 실패 작업 완전 실패 Slack으로 대체 알림
데이터 변환 오류 모든 데이터 손실 원본 데이터 유지

🔧 Try/Catch 실전 활용 패턴

실무에서 자주 사용되는 Try/Catch 구성 패턴을 익혀보겠습니다
🌐 API 호출 보호
Try: HTTP Request (외부 API 호출) ↓ Catch: Set 노드 (기본값 설정) ↓ Slack 알림
  • 외부 서비스 의존성 해결
  • 네트워크 오류 대응
  • 인증 실패 처리
🔄 데이터 변환 보호
Try: 복잡한 JSON 변환 ↓ Catch: 로그 기록 ↓ 원본 데이터 전달
  • JSON 파싱 오류 방지
  • 데이터 형식 변경 대응
  • 필수 필드 누락 처리
📧 서비스 연동 보호
Try: Gmail 발송 ↓ Catch: Slack 대체 알림 + 재시도 큐 추가
  • 이메일 발송 실패 대응
  • API 한계 초과 처리
  • 대체 알림 경로 확보
베스트 프랙티스: 외부 서비스를 사용하는 모든 노드에는 Try/Catch를 적용하여 안정성을 확보하세요.

🔬 고급 디버깅 기법

복잡한 워크플로우에서 정확한 문제 지점을 찾는 전문적인 디버깅 방법입니다
🔍 단계별 실행 및 분석
  1. 수동 단계별 실행
    • 각 노드 개별 실행
    • 중간 결과 상세 검토
    • 데이터 변환 과정 추적
  2. 예상 vs 실제 비교
    • 기대값과 실제값 대조
    • 데이터 타입 검증
    • 누락된 필드 확인
🛠️ 디버깅 전용 노드 활용
노드 활용 목적 사용법
Set 노드 중간 데이터 확인 로그 기록 및 변수 저장
IF 노드 조건부 실행 제어 특정 조건에서만 진행
Stop and Error 의도적 중단 문제 지점에서 정지

📊 프로덕션 모니터링 시스템

실시간으로 워크플로우 상태를 감시하고 문제를 조기에 발견하는 시스템입니다
🔔 알림 중심 모니터링
알림 유형 발생 조건 알림 방법
워크플로우 실패 즉시 Slack + 이메일
실행 시간 초과 30초 이상 Slack 경고
API 연결 실패 3회 연속 실패 담당자 직접 호출
데이터 이상 예상 범위 벗어남 이메일 리포트
🔄 알림 워크플로우 구성
워크플로우 실행 ↓ (실패시) Try/Catch 에러 감지 ↓ Set 노드 (에러 정보 정리) ↓ Slack 노드 (즉시 알림) + Gmail 노드 (이메일 백업)
  1. 에러 정보 수집 - 시간, 노드명, 오류 내용
  2. 우선순위 분류 - 긴급, 보통, 낮음
  3. 적절한 채널 선택 - Slack, 이메일, SMS

🔄 사후 분석 및 지속적 개선

문제 해결 후 재발 방지와 시스템 개선을 위한 체계적 접근법입니다
🎯 5단계 문제 해결 프로세스
  1. 즉시 대응
    • 알림 수신 즉시 확인
    • 실행 이력에서 에러 분석
  2. 원인 파악
    • 핀 고정으로 에러 재현
    • 근본 원인 분석
  3. 임시 해결
    • Try/Catch로 서비스 복구
    • 수동 처리로 긴급 대응
  4. 근본 해결
    • 워크플로우 구조 수정
    • 데이터 검증 로직 추가
  5. 재발 방지
    • 유사 상황 대비 로직 구현
    • 모니터링 강화
