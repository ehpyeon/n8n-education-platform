🔧 디버깅 팁 및 유지보수

문제 해결 전문가를 고용해서 완벽한 자동화 시스템을 만듭니다

  1. 실제 에러 사례 분석 - 고객 문의, 매출 데이터, 주문 처리 오류 해결
  2. 실행 이력 추적 기법 - 문제 발생 지점을 정확히 찾는 방법
  3. 핀 고정 디버깅 - 동일 조건에서 반복 테스트하는 기술
  4. Try/Catch 에러 방지 - 워크플로우가 멈추지 않도록 보호하는 방법

🔍 디버깅 접근법 개요

디버깅을 테스트와 에러 처리 두 가지 방법으로 나누어 체계적으로 접근합니다
🧪 1. 테스트 방법
  • Manual Trigger 활용 - 클릭, 챗으로 간편 테스트
  • 시작 지점 설정 - 특정 부분만 집중 테스트
  • 핀 고정 기능 - 동일 데이터로 반복 테스트
  • 단계별 실행 - 노드별 개별 검증
🛡️ 2. 에러 처리 방법
  • Continue on Error - 에러 후에도 계속 진행
  • 개별 노드 Error 분기 - Try/Catch 패턴
  • 글로벌 Error 캐치 - 전체 워크플로우 보호
  • IF 조건부 실행 - 특정 조건에서만 실행
핵심 원칙: 문제를 예방하는 것이 발생 후 해결하는 것보다 효율적입니다.

📋 1. Manual Trigger 활용법

클릭과 챗 기능으로 빠르고 간편하게 워크플로우를 테스트할 수 있습니다
💆 클릭 테스트
  1. 테스트 버튼 클릭
    • 워크플로우 에디터 상단 "Test workflow" 클릭
    • 즉시 전체 워크플로우 실행
    • 단발성 테스트에 적합
  2. 개별 노드 테스트
    • 특정 노드 선택 후 오른쪽 화살표 클릭
    • 해당 노드부터 끝까지 실행
    • 문제 구간 집중 테스트
💬 챗 테스트
상황 사용법 장점
챗봇 테스트 챗 노드로 가서 챗 전송 실제 사용자 경험
대화형 테스트 다양한 메시지 전송 여러 시나리오 확인
임시 데이터 입력 챗으로 테스트 데이터 전송 실제 데이터 없이 테스트
사용 팁: 여러 번 테스트해야 하는 경우 Manual Trigger를 선호하세요. 외부 시스템에 영향을 주지 않습니다.

🎯 2. 시작 지점 설정법

긴 워크플로우에서 특정 부분만 집중적으로 테스트할 수 있는 효율적인 방법입니다
🚀 시작 지점 설정 방법
  1. 테스트하고 싶은 노드 선택
    • 문제가 의심되는 노드 클릭
    • 오른쪽 패널에서 "여기서 시작" 옵션 확인
  2. 부분 실행 수행
    • 선택한 노드부터 끝까지만 실행
    • 전체 워크플로우 실행 시간 단축
  3. 결과 확인 및 반복
    • 각 노드의 출력 데이터 검증
    • 문제 해결까지 반복 테스트
📄 활용 시나리오
상황 시작 지점 장점
데이터 처리 오류 Set 노드 초기 데이터 수집 시간 절약
API 응답 문제 HTTP Request 다음 노드 외부 API 호출 없이 테스트
발송 시스템 오류 이메일/Slack 노드 데이터 처리 과정 스킵
효율성 팁: 긴 워크플로우에서는 반드시 시작 지점을 설정하여 테스트 시간을 단축하세요.

📌 3. 핀 고정 데이터 기능

동일한 데이터 조건에서 반복 테스트할 수 있는 강력한 디버깅 도구입니다
📌 핀 고정 사용법
  1. 실행 이력에서 데이터 선택
    • Executions 탭에서 적절한 실행 이력 선택
    • 특정 노드의 출력 데이터 확인
  2. 핀 고정 실행
    • 노드 위에 나타나는 핀 아이콘 클릭
    • "데이터 고정됨" 메시지 확인
  3. 고정 데이터로 테스트
    • 다음 노드부터 계속 실행
    • 동일 조건에서 반복 테스트
✨ 핀 고정의 장점
상황 기존 방식 핀 고정 방식
데이터 처리 오류 매번 전체 실행 문제 부분만 반복
API 응답 변경 실제 API 호출 다시 고정된 응답으로 테스트
발송 시스템 테스트 실제 메일 발송 안전한 테스트 환경
주의사항: 핀 고정된 데이터는 실제 실행에는 영향을 주지 않으니 테스트 완료 후 핀 해제를 잊지 마세요.

🔄 1. Continue on Error 설정

에러가 발생해도 워크플로우가 중단되지 않고 계속 진행되도록 설정하는 방법입니다
⚙️ Continue on Error 설정법
  1. 노드 설정 열기
    • 대상 노드 더블클릭
    • "Settings" 탭 선택
  2. Continue on Fail 옵션 활성화
    • "Continue on Fail" 체크박스 활성화
    • 에러 발생 시 다음 노드로 진행
📋 활용 사례
상황 기존 방식 Continue on Error
API 연결 실패 워크플로우 중단 다음 단계로 진행
이메일 발송 실패 전체 작업 중단 대체 알림 시도
데이터 변환 오류 처리 중단 원본 데이터 유지
주의사항: Continue on Error를 사용할 때는 에러 상황을 다음 단계에서 적절히 처리하도록 설계하세요.

🛡️ 2. 개별 노드 에러 분기 설정

노드에서 에러가 났을 때 분기 치는 설정을 해놓으면, 코딩의 Try & Catch와 비슷하게 개별 노드마다 에러 처리 분기를 만들 수 있습니다
🔄 에러 분기 설정 방법
  1. 에러 가능성 있는 노드 선택
    • HTTP Request, 이메일 등 외부 서비스 노드
    • 데이터 변환, JSON 파싱 노드
  2. 에러 분기 연결
    • 노드 아래쪽 빨간 점에서 연결선 끌어서 연결
    • 에러 발생 시에만 실행되는 별도 경로 생성
  3. 에러 처리 로직 구성
    • 에러 정보 로그 기록
    • 대체 방법 시도 (다른 API, 기본값 등)
    • 관리자에게 알림 전송
📊 에러 분기 구조 예시
[HTTP Request 노드] │ (정상 실행) ↓ [Set 노드 - 데이터 처리] │ ↓ [Slack 성공 알림] [HTTP Request 노드] │ (에러 발생) ↓ (빨간 연결선) [Set 노드 - 에러 정보 기록] │ ↓ [Slack 에러 알림 + 대체 작업]
코딩과의 유사점: 프로그래밍의 Try-Catch 구문과 동일한 개념으로, n8n에서는 시각적으로 에러 처리 흐름을 구성할 수 있습니다.

🌐 3. 글로벌 Error 캐치 설정

워크플로우 전체에 에러 처리를 적용하여 시스템 전체를 보호하는 방법입니다
⚙️ 글로벌 Error 캐치 설정
  1. 워크플로우 설정 열기
    • 워크플로우 이름 옷의 설정 아이콘 클릭
    • "Settings" 또는 "설정" 단추 선택
  2. Error Workflow 연결
    • "Error Workflow" 옵션 찾기
    • 에러 처리용 별도 워크플로우 선택
  3. 에러 처리 워크플로우 제작
    • 에러 정보 수집 노드
    • 알림 전송 노드
    • 로그 기록 노드
📋 글로벌 Error 워크플로우 구성
Error Trigger ↓ [Set 노드] - 에러 발생 시간 - 워크플로우 이름 - 에러 메시지 - 실패한 노드 이름 ↓ [Slack 노드] - 즉시 알림 + [Gmail 노드] - 이메일 백업
주요 장점: 개별 Try/Catch를 설정하지 않아도 모든 에러를 중앙에서 통합 처리할 수 있습니다.

🤔 4. IF 조건부 실행 제어

특정 조건에서만 실행되도록 하여 불필요한 에러를 예방하는 방법입니다
🔀 IF 노드 활용법
  1. 데이터 유효성 검사
    • 빈 값 체크: {{ $json.data !== null }}
    • 문자열 길이 확인: {{ $json.name.length > 0 }}
    • 숫자 범위 검사: {{ $json.age > 0 && $json.age < 120 }}
  2. 비즉니스 시간 제한
    • 주말 제외: {{ moment().day() !== 0 && moment().day() !== 6 }}
    • 밤 시간 제한: {{ moment().hour() >= 9 && moment().hour() < 18 }}
📊 조건부 실행 예시
상황 조건 효과
이메일 발송 이메일 주소 유효성 잘못된 주소로 인한 에러 방지
API 호출 네트워크 연결 상태 오프라인 상태에서 호출 방지
대용량 처리 데이터 크기 확인 메모리 부족 에러 예방
예방 원칙: 에러가 발생할 수 있는 상황을 사전에 차단하여 안정성을 높이세요.

📊 실제 사례: 고객 서비스 시스템 에러

디버깅 방법들을 종합적으로 활용한 실제 문제 해결 사례를 살펴보겠습니다
🔍 문제 상황
항목 내용
증상 Slack 알림이 오지 않음
원인 API 응답 형식 변경
피해 6시간 동안 147건 미처리

🛠️ 종합 해결 과정
  1. Manual Trigger로 빠른 테스트
    • 챇 기능으로 즉시 워크플로우 테스트
    • 문제 현상 재현 확인
  2. 핀 고정으로 정확한 재현
    • 정상 작동했던 데이터 핀 고정
    • 동일 조건에서 에러 다시 재현
  3. Try/Catch로 예방 체계 구축
    • HTTP Request에 Try/Catch 추가
    • 에러 시 기본값으로 대체 처리
  4. IF 조건으로 사전 차단
    • API 응답 구조 검증 로직 추가
    • 예상 형식이 아니면 알림 전송
핵심 교훈: 한 가지 방법보다는 여러 디버깅 기법을 조합하여 사용할 때 더 효과적입니다.

🛡️ 예방 시스템 구축 전략

에러가 발생하기 전에 미리 차단하고 대비하는 종합적인 예방 전략입니다
🔄 다중 방어망 구성
  1. 1차 방어: IF 조건 검사
    • 데이터 유효성 사전 확인
    • 시스템 상태 검사
  2. 2차 방어: Try/Catch 보호
    • 외부 서비스 연동 보호
    • 대체 경로 준비
  3. 3차 방어: Continue on Error
    • 워크플로우 중단 방지
    • 서비스 연속성 보장
  4. 4차 방어: 글로벌 Error 캐치
    • 예상치 못한 에러 중앙 처리
    • 즉시 알림 및 로깅

모니터링 대상 감지 방법 알림 체계

예방 원칙: 에러를 발생 후 해결하는 것보다 사전에 차단하는 것이 모든 면에서 비용 효율적입니다.

🎯 디버깅 마스터 로드맵

효과적인 디버깅을 위한 체계적인 접근법과 실무 전략을 정리합니다
🧪 테스트 방법 체크리스트
  • Manual Trigger - 클릭/챗으로 빠른 테스트
  • 시작 지점 설정 - 특정 구간만 집중 테스트
  • 핀 고정 - 동일 데이터로 반복 테스트
  • 단계별 실행 - 각 노드 개별 검증
테스트 전략: 1. Manual Trigger로 대략적 확인 2. 시작 지점으로 문제 구간 추리 3. 핀 고정으로 정확한 재현 4. 단계별 실행으로 세밀 분석
🛡️ 에러 처리 체크리스트
  • Continue on Error - 노드별 에러 후 계속
  • Try/Catch - 개별 노드 에러 분기
  • 글로벌 Error 캐치 - 전체 워크플로우 보호
  • IF 조건부 - 사전 예방 차단
에러 처리 전략: 1. IF로 사전 차단 (예방) 2. Try/Catch로 개별 보호 3. Continue on Error로 연속성 4. 글로벌 캐치로 최종 보호
마스터 비법: 문제를 예방하는 것이 발생 후 해결하는 것보다 10배 더 효율적입니다. 디버깅은 문제 해결 도구가 아니라 예방 도구입니다.
